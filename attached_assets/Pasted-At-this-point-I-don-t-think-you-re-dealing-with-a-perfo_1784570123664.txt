At this point, I don't think you're dealing with a performance problem anymore. I think you're dealing with a scan engine state problem.

From your screenshots, several things don't line up.

1. The indexer says "20 / 20 files" but never completes

That is the biggest clue.

A healthy scan should go something like:

Loading database...
Walking...
Processing...
Writing DB...
Completed

Instead yours is stuck at:

Indexing
20 / 20 files
100%

…and it never exits.

That usually means one of these:

a worker thread never signaled completion
the DB batch writer never flushed
a Promise never resolved
the job state machine never transitioned to COMPLETE
a queue thinks work is still pending even though there isn't any

This is not an SMB speed problem.

2. Dashboard says scanning forever

Your dashboard and header are both still showing scanning.

That means the server still believes:

activeJob != null

or

status == RUNNING

even though the scan appears finished.

So something is holding the scan open.

3. People page returns HTTP 500

That is unrelated to SMB.

A 500 means the backend crashed handling the request.

Possibilities include:

media_ai table issue
face table missing
migration incomplete
query timeout
null reference
connection already locked

This needs its own fix.

4. Optimize immediately fails

Again, not an SMB issue.

If Optimize fails instantly, then either:

it can't create a job
the job engine is locked
another job never released its lock
the job scheduler thinks something is still running

That lines up with the scan never completing.

What I suspect is actually happening

Everything points back to the Job Engine.

Imagine this:

Scan starts

↓

loads DB

↓

walks 20 directories

↓

workers finish

↓

DB writes finish

↓

activeWorkers = 0

↓

queue empty

↓

completion callback never fires

↓

job stays RUNNING forever

Now every other subsystem waits forever.

That explains:

scanning forever
optimize failing
dashboard hanging
resume/discard weirdness
thumbnails not progressing
