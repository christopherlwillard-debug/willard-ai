Same idea for Force Discard

I like the feature.

But I'd slightly change the acceptance criteria.

Instead of only:

Force discard clears stuck jobs.

I'd require it to verify cleanup:

active job removed
queue closed
worker pool drained
timers cancelled
event listeners removed
DB lock released
next scan starts successfully

Otherwise it risks becoming a "clear the symptom" button while leaving resources orphaned.

Overall

I actually think this is the point where your project should pivot from feature development to a short stabilization milestone.

If I were managing this roadmap, I'd temporarily freeze #123–#151 and create a dedicated milestone:

S1 — Instrumentation
S2 — Scan lifecycle stability
S3 — Job recovery
S4 — Error reporting
S5 — Performance optimization (resume roadmap)

That order is likely to save a lot of time because you'll stop guessing where the failures occur and start fixing verified causes.
