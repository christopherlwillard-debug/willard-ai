#172 - Fix Global "Scanning Forever" State
What & Why
The app shows "Scanning…" in the header indefinitely — even hours after a scan completes. This causes Collections, Library, People, Documents, Optimize, Cleanup, and AI Search to show spinners or empty states because they depend on the same global scan flag.

Two bugs were confirmed by direct code inspection:

Confirmed Bug #1 — Global scan state reads from the legacy job table routes/dashboard.ts computes isScanning by querying scanJobsTable for status "running" (lowercase). The current library engine writes all jobs to libraryJobsTable with status "RUNNING" (uppercase). These are two separate parallel scan systems. The dashboard never reads the current engine's state, so any old scanJobsTable row stuck in "running" will freeze isScanning: true for the lifetime of the server — indefinitely.

Confirmed Bug #2 — Dashboard performs synchronous NAS I/O on every poll The dashboard route calls synchronous checkNasReachable(nasPath) (which executes existsSync + statSync + accessSync + readdirSync on the NAS path) and a synchronous getDiskStats() on every request. The dashboard is polled every 3 seconds while isScanning: true. On a Windows NAS (Z:) each of these calls can stall for hundreds of milliseconds, blocking the Node.js event loop and making every other concurrent API request (Collections, People, etc.) appear to hang.

Potential Effect The combination of Bug #1 keeping the poll interval locked at 3 seconds and Bug #2 blocking the event loop on each poll explains why every page in the app appears frozen while a scan is nominally running.

Underlying architectural drift scan.ts (legacy) and job-engine.ts (current) are both fully wired up, writing to different tables (scanJobsTable vs libraryJobsTable) with different status enums. The dashboard watches only the legacy table. Retiring or clearly separating the two systems would eliminate this class of bug permanently.

Done looks like
Before any code changes: a startup log clearly states how many jobs are in each table and what the computed isScanning value is and why (confirming which bug is active)
After a successful scan, the header returns to "Full Rescan" within one poll cycle (≤ 30 seconds)
Collections, Library, People, Documents, and Cleanup load data immediately without waiting for a scan to finish
Navigating the app while a scan is running remains responsive
There is exactly one authoritative source of truth for scan state — the UI, dashboard, and job engine all derive isScanning from libraryJobsTable exclusively, with no other table or variable shadowing it
Out of scope
Replacing or removing the legacy scan.ts / scanJobsTable system entirely (investigate and document, but do not delete until routes are audited)
AI Search embedding analysis (0/24551 files) — separate enrichment feature
Steps
Instrument first — log the exact state before touching anything — On server startup (and on each GET /api/dashboard request in debug mode), emit a structured log showing: count of libraryJobsTable rows in each status, count of scanJobsTable rows in each status, and the final isScanning value with the reason. This proves which bug is causing the frozen header before any fix is applied.

Fix isScanning source in dashboard — Change the dashboard route to derive isScanning from libraryJobsTable status "RUNNING" (covering SCAN, THUMBNAILS, and METADATA job types) and remove the scanJobsTable query from this computation entirely.

Drain stale scanJobsTable legacy jobs at startup — Add a startup cleanup step (alongside the existing libraryJobsTable RUNNING→FAILED sweep) that marks any scanJobsTable rows stuck in "running" as "failed", so a restart always clears the stale state.

Make dashboard NAS checks async — Replace the synchronous checkNasReachable + getDiskStats calls in routes/dashboard.ts with the existing checkNasReachableAsync. Cache the result briefly (e.g. 5 seconds) so high-frequency polls during a scan don't re-stat the NAS on every request.

Convert remaining sync checkNasReachable call sites — Audit and convert the remaining synchronous call sites in health.ts, system.ts, app.ts, ai-enrichment.ts, and face-recognition.ts to use checkNasReachableAsync.

Audit scanJobsTable usage and document retirement path — List every route and UI component that still reads or writes scanJobsTable. Determine whether the legacy /scan, /scan/status, and /scan/history routes are still called by the frontend. If they are unused, mark them deprecated with a comment block. Do not delete them in this task, but leave a clear note for the next task that does.

Verify pages load independently of scan state — Check that Collections, People, Documents, and Cleanup fetch data unconditionally from the DB. If any is found to gate loading on isScanning, remove that gate.
